React.useEffect(() => {
  if (open) {

    );

function handleDelete() {

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User → ClientDialog form | nom, adresse, telephone, email entered by user, validated by zod before store mutation |
| User → delete AlertDialog | Confirms deletion of a customer by id; id comes from in-memory store (no external injection) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-03 | Tampering | ClientDialog form inputs | mitigate | All fields validated by Zod schema before store write. `nom` min 2, `adresse` min 5, `telephone` min 1, `email` valid format or empty. `maxLength` on Input elements provides DOM-level cap. Values are `.trim()`-ed before store write. |
| T-06-04 | Tampering | deleteCustomer via AlertDialog | accept | Customer id comes from `customers` array rendered from store. No external UUID injection possible through the UI. Alert requires explicit user confirmation before mutation fires. |
| T-06-05 | Information Disclosure | Customer data in localStorage | accept | Inherited from Phase 2 risk posture. Customer data (nom, adresse, telephone, email) is POC demo data, not real PII in this context. |
| T-06-06 | Tampering | email field | mitigate | `z.string().email()` validates format when present. `type="email"` on the Input provides browser-level format hint. Empty string maps to `undefined` before store write (no malformed email persisted). |
</threat_model>
